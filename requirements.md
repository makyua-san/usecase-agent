## 1. 実行時間の考え方

これはその設計でよいです。むしろ自然です。

### 推奨

* **ソフト制限: 20分**

  * エージェントへの指示に含める
  * 20分以内で価値の高いところまで終える意識を持たせる
  * 20分を超えそうなら、途中成果の保存と次回への引き継ぎを優先
* **ハード制限: 30分**

  * ハーネスが強制終了
  * 終了時は run 状態を `timeout` にする
  * 部分成果と途中ログを確実に残す

これでよい理由は、LLM系は「少し延ばせばまとまる」ケースがある一方、無制限にすると運用崩壊するからです。
20分で自律的に切り上げる文化を作りつつ、30分で安全装置をかけるのがちょうどいいです。

## 2. 実行モードは「前回の結果から決まる」が正しい

ここはかなり重要です。
おっしゃる通り、「次どのモードで行くべきか」を判断する層が必要です。

ただ、その判断はその場の思いつきではなく、**前回 run の成果物として残す** のがよいです。

つまり 1 run の最終成果物は、収集結果だけでは足りません。
最低でも次の4つを残すべきです。

* 今回の成果
* 今回の課題
* 次回やるべきこと
* 推奨モード

---

## 3. モード設計はこのくらいが扱いやすい

今出ている「コンテキスト確認、タスク整理、探索、ネクストステップブラッシュアップ」はかなり良いです。
少し整理すると、こうなります。

### 推奨モード

#### 1) Context Review

* 既存知識の確認
* 直近 run の失敗確認
* 未処理タスク確認
* 重点テーマ確認

#### 2) Task Organize

* 今回の探索テーマ決定
* 優先順位付け
* クエリ戦略決定
* 情報源配分決定

#### 3) Explore

* Playwright で実際に探索
* 候補の取得
* 必要に応じて深掘り
* 一次抽出

#### 4) Consolidate

* 重複判定
* 正規化
* canonical case への統合
* スコア更新

#### 5) Next-Step Design

* 次回探索の仮説作成
* 失敗要因整理
* 次回の推奨モード決定
* 継続タスク発行

これならかなり自然です。

## 4. 「次回の脳」は専用の軽い planner にするのが良い

ここで1つ重要なのは、
探索本体のエージェントと、次回方針を決める planner を分けるかどうかです。

私は、**最初は分けなくてもよいが、成果物として planner 的な出力を必須化する** のが良いと思います。

つまり最初は同じエージェントでもよいです。
ただし run 終了時に必ずこういうJSONを出させます。

```json
{
  "recommended_next_mode": "Explore",
  "recommended_focus": [
    "国内の製造業における生成AI活用事例",
    "LTや登壇資料由来の実践ノウハウ"
  ],
  "open_questions": [
    "導入済み事例と検討中事例の区別が曖昧",
    "同一事例の転載記事が多い"
  ],
  "search_improvements": [
    "企業広報よりイベント資料を優先",
    "英語圏は 'workflow', 'ops', 'copilot', 'playbook' で検索強化"
  ],
  "pending_tasks": [
    "case_028 の続報確認",
    "重複候補3件の統合判定"
  ]
}
```

これがあると、次回起動時の判断材料になります。

## 5. エラーハンドリングは最初からかなり丁寧に入れるべき

ここは同意です。
特に使用制限やレート制限は確実に起きます。

### 最低限必要なエラー分類

#### A. リトライ不要

* 404
* 明らかな構文エラー
* robots / アクセス禁止
* unsupported page

→ そのままスキップして記録

#### B. 短期リトライ

* 一時的タイムアウト
* Playwrightのページロード失敗
* 一時的な接続不良

→ 数回リトライ、だめならスキップ

#### C. クールダウンが必要

* rate limit
* usage limit
* provider quota
* anti-bot的挙動

→ run内ではそのソースを止める
→ 次回以降の探索戦略に反映

#### D. run継続不能

* DBロック異常
* ハーネス例外
* 状態破損
* 永続化失敗

→ runを fail 扱いにして安全終了

### 特に大事な挙動

* **使用制限に引っかかったら全体停止しない**
* **その provider / source を quarantine 扱いにしてスキップ**
* **次回 planner に「この制限を避ける探索方針」を残す**

たとえば:

* 公式サイトに直接行きすぎた
* 同一ドメインへ短時間アクセスしすぎた
* Playwright の挙動が重すぎた

こういう反省を構造化して残すべきです。

## 6. Playwright 前提はあり。ただし「ブラウザを振り回しすぎない設計」にする

Playwright前提でもよいです。
ただし、すべてを重いブラウザ操作で処理すると非効率です。

なので思想としてはこうです。

### Playwright前提の現実的な運用

* Playwright を**標準収集手段**にする
* ただし毎回フルブラウザ的に重く使わない
* 一覧ページ取得、DOM抽出、遷移、スクリーンショット取得を中心にする
* 動的レンダリングやクリック展開が必要なときだけ深く使う

つまり、
**Playwrightを持つが、操作は段階的にする**
という設計がよいです。

たとえば取得深度を3段階で持つとよいです。

* depth 1: 一覧取得だけ
* depth 2: 詳細ページ閲覧
* depth 3: 複数遷移やUI操作

これで負荷制御しやすくなります。

## 7. マルチエージェント拡張は面白い。ただし最初から「共有状態の規律」を入れる

ここもその通りです。
マルチにすると急に難しくなるのは、知能より**状態管理**です。

最初から必要なのは次の3つです。

### 1) task ownership

どのエージェントがどのタスクを担当中か明示する

### 2) lock / lease

同じケースやソースを複数エージェントが重複処理しないようにする

### 3) context scope

各エージェントが見てよいコンテキスト範囲を明示する

---

### マルチエージェント時の役割例

* **Planner**

  * 次回方針決定
* **Scout**

  * 候補探索
* **Reader**

  * 詳細読解
* **Normalizer**

  * 構造化
* **Deduper**

  * 重複統合
* **Reviewer**

  * 品質チェック

ただ、MVPではまだ多いです。
最初は単一エージェント + ハーネスで十分です。
将来に備えてDBスキーマだけ分離可能にしておくとよいです。

## 8. DBはまず SQLite でかなり妥当

結論として、**最初は SQLite でよい** です。
むしろかなり相性がいいです。

### SQLite が向いている理由

* 単一サーバー前提なら十分
* 追加コストがない
* バックアップが簡単
* ローカルファイルとして扱いやすい
* 初期実装がかなり速い
* ダッシュボードからも触りやすい

### SQLite の弱点

* 高並列書き込みは弱い
* 分散構成には向かない
* ジョブが増えすぎると調整が必要

ただ、今の構想だと
「1時間に1回」「基本単一run」「将来拡張あり」なので、かなり合っています。

---

### おすすめ構成

* **SQLite**

  * run
  * task
  * case
  * source
  * reflection
  * feedback
  * lock
* **ファイル保存**

  * logs/
  * raw_html/
  * screenshots/
  * extracted_json/

これが一番バランスがいいです。

## 9. ノイズ対策は「何をユースケースとして採用するか」の定義を厳しくするのが先

ここは重要です。
ユーザーが欲しいのは「生成AIを使うとき、使い方、ノウハウがわかる例」ですよね。
ならば、収集対象の定義をニュース一般ではなく、**実践知があるもの** に寄せるべきです。

### ユースケース採用条件の例

以下のうち2つ以上を満たす場合に採用、のようにするとよいです。

* 誰が使っているか明確
* 何に使っているか明確
* ワークフローが少しでもわかる
* 導入効果や改善点が示唆されている
* 失敗や工夫が書かれている
* ツール構成や実装のヒントがある
* 登壇資料やLTで実運用の匂いがある

逆に、こういうものはノイズ寄りです。

* 単なる「生成AIを活用します」宣言
* 抽象論だけの広報文
* 実際の業務プロセスが見えない記事
* 流行語だけを並べたまとめ

## 10. 収集対象は「実践知レベル」で段階化するとよい

ニュース、プレスリリース、LT、発表資料を全部混ぜると粗さが違いすぎます。
なので、収集時にレベルを持たせると使いやすいです。

### 例

* **Level A: 実運用ノウハウあり**

  * LT
  * 導入ブログ
  * スライド
  * 技術登壇
  * 事例記事
* **Level B: 活用内容は見える**

  * プレスリリース
  * 事例紹介ページ
* **Level C: シグナルのみ**

  * 一般ニュース
  * 短い告知
  * イベント紹介

このレベルを持っておくと、ダッシュボードでも「実践寄りだけ見たい」ができます。

### ブラッシュアップ後の中核コンセプト

* 非対話エージェントが定期的に国内外の生成AIユースケースを探索
* Playwright を基本手段として情報収集
* ハーネスがモード、時間、実行環境、エラー、ログを統制
* 各runの最後に、次回の探索方針と改善点を構造化して残す
* ダッシュボードから結果、ログ、改善提案、FBを確認・制御
* 初期DBは SQLite、証跡はファイル保存
* 将来はマルチエージェント化できるよう task / lock / context scope を最初から持つ
